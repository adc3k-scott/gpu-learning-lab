Domain 1: System & Server Bring-Up (31%)

Deployment sequences, network topology, BMC/OOB, firmware, GPU install, cabling

1
What is the deployment sequence for a DGX SuperPOD?
▾
1. Physical install (racks, power, cooling, cabling) → 2. OOB network + BMC config → 3. In-band management network → 4. Head node / BCM install → 5. Compute node provisioning (PXE boot from BCM) → 6. Storage fabric + parallel FS → 7. Compute fabric (InfiniBand) → 8. GPU driver + CUDA install → 9. Cluster validation (HPL/NCCL) → 10. Workload manager (Slurm/K8s)
2
What are the four network fabrics in a SuperPOD?
▾
1. Compute Fabric — InfiniBand (Quantum-X800 XDR 800G or QM9700 NDR 400G). GPU-to-GPU traffic.
2. Storage Fabric — InfiniBand (QM9700 NDR) or Ethernet (Spectrum-4 SN5610 800G). Data I/O.
3. In-Band Management — Ethernet (SN5610/SN5600D, 64x 800G). OS-level management.
4. Out-of-Band Management — Ethernet (SN2201, 48x 1G + 4x 100G). BMC/IPMI traffic.
3
What is BMC and how is it configured?
▾
Baseboard Management Controller — embedded controller for out-of-band management. Works even when OS is crashed.
Functions: Remote power on/off/reset, KVM console, health monitoring, firmware updates, event logging, serial-over-LAN.
Protocol: IPMI (legacy) or Redfish (modern REST API).
Network: Dedicated OOB network (SN2201 switches), separate from compute/storage.
Config: Static IP on OOB subnet, LDAP/AD for auth, SNMP for alerts.
4
What is Redfish and why does it replace IPMI?
▾
Redfish = modern REST API for server management (replaces IPMI).
Why better: RESTful (JSON over HTTPS), secure (TLS), scriptable, supports modern auth, standardized schema (DMTF).
IPMI problems: Plain text, weak auth, limited extensibility, no encryption by default.
Both access BMC, but Redfish is the modern standard for DGX management.
5
What is TPM and why is it important for DGX?
▾
Trusted Platform Module — hardware security chip on motherboard.
Functions: Secure boot chain verification, disk encryption key storage, hardware-based attestation, firmware integrity measurement.
For DGX: Ensures only authorized firmware/OS boots. Critical for sovereign AI, defense, SCIF deployments. Prevents tampering at hardware level.
6
What is a rail-optimized fat-tree topology?
▾
Each GPU in a DGX has 8 NICs (one per GPU = one "rail"). Rail-optimized means GPU 0 from every node connects to the same set of leaf switches, GPU 1 to another set, etc.
Why: AllReduce operations in training follow rail patterns — GPU 0 talks to GPU 0 on other nodes. Rail-optimized topology ensures this traffic stays within one leaf group, minimizing spine hops.
Result: Non-blocking bandwidth for the most common collective communication patterns.
7
DGX H100 SuperPOD: how many nodes per Scalable Unit?
▾
1 SU = 32 DGX H100 nodes (one reserved for UFM = 31 usable).
4 nodes/rack × 8 racks = 32 nodes = 256 GPUs per SU (248 usable).
Reference design: 4 SU = 127 nodes = 1,016 GPUs.
IB fabric per SU: 8 leaf switches + 4 spine switches.
Power: ~40 kW/rack, 10.2 kW per DGX H100.
8
DGX B300 SuperPOD: how many nodes per Scalable Unit?
▾
1 SU = 72 DGX B300 nodes = 576 GPUs.
Reference design: 8 SU = 576 nodes = 4,608 GPUs.
Max scale: 72+ SU, 2,000+ nodes.
~56 kW/rack. 4 DGX B300 per rack (2U each).
Uses Quantum-X800 (XDR 800G) switches.
3x 2U rack PDUs per rack for redundancy.
9
GB200 NVL72 rack specifications?
▾
72 Blackwell GPUs + 36 Grace CPUs in a single rack.
Power: 120-140 kW per rack (GB200) / ~132 kW (GB300). Liquid cooling mandatory.
Coolant: 25°C inlet, +20°C delta, up to 130 LPM flow rate.
NVLink 5: 1,800 GB/s per GPU. NVSwitch 4: 72 ports.
Aggregate bandwidth: 130 TB/s.
All 72 GPUs connected all-to-all via NVSwitch — acts as single giant GPU.
GB300 upgrade: B300 GPUs (288 GB HBM3e each = 21 TB total vs 13.5 TB), 1.1 ExaFLOPS FP4, ConnectX-8 with integrated PCIe switch.
Grace CPUs connect to GPUs via NVLink-C2C at 900 GB/s — 7x faster than PCIe Gen5.
10
InfiniBand generations: HDR vs NDR vs XDR?
▾
HDR: 200 Gb/s per port, PAM4 50G signaling, QSFP56 connector (2019).
NDR: 400 Gb/s per port, PAM4 100G, OSFP twin-port (2022). Quantum-2 switch: 64 ports.
XDR: 800 Gb/s per port, PAM4, OSFP (2025). Quantum-X800 switch: 144 ports, 14.4 TFLOPS in-network compute (SHARP).
Each generation doubles port speed.
11
ConnectX-7 vs ConnectX-8 NIC specs?
▾
ConnectX-7: NDR 400 Gb/s InfiniBand, OSFP/QSFP112, PCIe Gen5. Used in DGX H100.
ConnectX-8: XDR 800 Gb/s InfiniBand — first 800G SuperNIC. Dual-protocol (IB or Ethernet). Single-port OSFP (800G IB) or dual-port (2x 400G Ethernet). PCIe Gen6 x16. Used in DGX B300.
12
Cable types for InfiniBand: DAC vs AOC vs Transceivers?
▾
DAC (Direct Attach Copper): Short runs (up to ~3m), lowest latency, lowest cost, passive. Best for intra-rack.
AOC (Active Optical Cable): Medium distance (up to ~100m), fixed connectors, good for inter-rack within a row.
Transceivers + Fiber: Longest runs (100m-2km+), swappable optics, highest cost. Required for spine-to-leaf in large fabrics.
NDR uses OSFP twin-port cages. XDR uses OSFP single-port.
13
DGX H100 power and PDU configuration?
▾
DGX H100: 10.2 kW peak, 6 PSUs (min 4 required), 200-240 VAC.
Rack: 4 DGX per rack = ~40 kW/rack.
PDU config: 3 separate rPDU sources per rack minimum. N+1 redundancy. Each source sized for 50% of peak.
Preferred: 415 VAC, 32A, 3-phase Wye = 21.8 kW per circuit.
Phase balancing: each PSU on different phase leg, 2-rack pattern.
Max 2 vertical (0U) PDUs; remaining horizontal.
Vendors: Raritan, Vertiv/Geist, ServerTech.
14
Power per system across DGX generations?
▾
DGX H100: 10.2 kW (air cooled, ~40 kW/rack)
DGX B200: ~14.3 kW (air or liquid)
DGX B300: ~14 kW (~56 kW/rack, liquid required)
GB200 NVL72: 120-140 kW (full rack, liquid mandatory)
Future Rubin: 250-900 kW/rack projected (liquid mandatory)

Cooling methods by rack density?
▾
Air cooling: up to ~30 kW/rack, PUE 1.3-1.5. DGX H100 standard.
Rear-Door HX (passive): 30 kW, PUE 1.2-1.3. Retrofit existing facilities.
Rear-Door HX (active): 75 kW, PUE 1.1-1.2. Medium density.
Direct Liquid / CDU: 140+ kW, PUE 1.05-1.1. DGX B200, GB200 NVL72.
Single-phase immersion: 200+ kW, PUE 1.02-1.03. EC-110 fluid.
Two-phase immersion: 200+ kW, PUE 1.01. Ultra-high density.
ASHRAE A1 envelope: 18-27°C inlet air.
16
CDU specifications for liquid cooling?
▾
CDU = Coolant Distribution Unit. Transfers heat from server loop to facility water.
Specs: 180/240/250 kW capacity options. Redundant PSU, dual hot-swap pumps.
GB200 NVL72: Coolant inlet 25°C, outlet 45°C (+20°C delta). Flow rate up to 130 LPM per rack.
Industry target: 1.5 L/min per kW for 10°C delta.
Blackwell: 300x better water efficiency vs air cooling.
QCT QoolRack: stand-alone liquid cooling for GB200.

NVLink generations and bandwidth?
▾
Gen 1 (Pascal): 80 GB/s per GPU, 4 links
Gen 2 (Volta): 300 GB/s, 6 links
Gen 3 (Ampere): 600 GB/s, 12 links
Gen 4 (Hopper): 900 GB/s, 18 links
Gen 5 (Blackwell): 1,800 GB/s, 18 links
Gen 6 (Rubin): 3,600 GB/s, 18 links
Each gen ~doubles per-link bandwidth. Link count stabilized at 18 since Hopper.

NVSwitch generations and port counts?
▾
Gen 1 (Volta): 18 ports, 50 GB/s each. DGX-2 (16 GPUs).
Gen 2 (Ampere): 36 ports, 50 GB/s. DGX A100 (8 GPUs, 4 switches).
Gen 3 (Hopper): 64 ports, 50 GB/s. DGX H100 (8 GPUs, 4 switches).

Storage requirements for SuperPOD?
▾
High-Performance Storage: InfiniBand or Ethernet attached, POSIX-compliant parallel FS.
Supported FS: Lustre (DDN EXAScaler), IBM Spectrum Scale (GPFS), WekaFS, BeeGFS — all GDS certified.

What firmware needs to be validated during bring-up?
▾
GPU firmware (InfoROM — GPU BIOS, ECC settings)

Domain 2: Cluster Test & Verification (33%)

HPL, NCCL benchmarks, NVLink verification, ClusterKit, burn-in, fabric validation

23
What is HPL and how is it used for cluster validation?
▾
High Performance Linpack — benchmark that solves dense system of linear equations. Measures peak FLOPS.
Purpose: Validates GPU compute + memory + interconnect under sustained load. If HPL performance matches expected specs, hardware is healthy.
DGX H100 expected: ~80% of theoretical peak per node.
Run: NGC container nvcr.io/nvidia/hpc-benchmarks. Scale from single node to full cluster.
Key params: N (matrix size), NB (block size), P×Q (process grid). Tune for >80% efficiency.
24
What are NCCL tests and what do they validate?
▾
NCCL Tests — benchmark suite for GPU collective communications.
Tests: all_reduce_perf, all_gather_perf, reduce_scatter_perf, broadcast_perf, sendrecv_perf.
Validates: NVLink bandwidth (intra-node), InfiniBand bandwidth (inter-node), NCCL topology detection, SHARP offload.
Key metric: Bus bandwidth (GB/s) — should match NVLink/IB specs.
DGX H100 expected: ~900 GB/s intra-node (NVLink), ~400 Gb/s inter-node (IB NDR).
Run: nccl-tests from NGC or build from GitHub. Test single-node first, then multi-node.
25
DCGM diagnostic levels (r=1 through r=4)?
▾
r=1 (Short): ~seconds. Tests: Software deployment, PCIe+NVLink, GPU Memory, Memory Bandwidth.
r=2 (Medium): <2 min. All r=1 + Diagnostics plugin.
r=3 (Long): <30 min. All r=2 + SM Stress, Targeted Stress, Targeted Power, EUD (~5 min).
r=4 (Extra Long): 1-2 hours. All r=3 + Memory Stress, Pulse Test, Memtest, extended EUD (~20 min).
Common use: r=1 for quick health check. r=3 for pre-job validation. r=4 for burn-in/acceptance.
dcgmi diag -r 3 # most common pre-job dcgmi diag -r 4 -j # burn-in + JSON output dcgmi diag -r pulse_test # specific test only
26
What are the 11 DCGM diagnostic plugins?
▾
1. Deployment — CUDA env, NVML, permissions, persistence mode, page retirement
2. Diagnostic — core GPU diagnostics
3. PCIe-GPU Bandwidth — h2d, d2h, p2p, latency
4. GPU Memory — memory subsystem validation
5. Targeted Power — power consumption analysis
6. Targeted Stress — compute load stress
7. Memtest — 11 test patterns (Test0-10), default runs Test7+Test10 for 600s
8. Pulse Test — current spike testing for PSU validation
9. NVBandwidth — bandwidth measurement
10. Memory Bandwidth — memory throughput
11. NCCL Tests — multi-GPU collective communication
27
What is ClusterKit and how is it used?
▾
ClusterKit = NVIDIA's cluster-level assessment tool. Runs automated checks across all nodes.
Validates: Firmware versions match across cluster, GPU health (DCGM), NVLink status, InfiniBand link status, cable quality, driver versions, CUDA compatibility.
Output: Pass/fail report per node with specific failure reasons.
Use case: Run after initial deployment and after any hardware change. Catches version mismatches and degraded links before workloads run.
28
Burn-in procedure for a new DGX cluster?
▾
1. Run DCGM r=4 on every node (1-2 hours each) — catches infant mortality
2. Run single-node NCCL tests — validate NVLink bandwidth per node
3. Run single-node HPL — validate per-node FLOPS
4. Run multi-node NCCL — validate IB fabric bandwidth
5. Run multi-node HPL at full scale — sustained load test (24-72 hours)
6. Monitor thermals, power, ECC errors throughout. Any uncorrectable ECC = RMA that GPU.
7. Run ClusterKit assessment — final validation report.
Pass criteria: >80% HPL efficiency, zero UCE, NVLink/IB bandwidth within spec.
29
How do you validate NVLink health?
▾
nvidia-smi nvlink -s # NVLink status summary nvidia-smi nvlink -e # error counters nvidia-smi nvlink -c # capabilities dcgmi diag -r 1 # includes NVLink check nccl-tests/build/all_reduce_perf # bandwidth validation
Healthy indicators: All links active, zero replay/recovery errors, bandwidth matches spec (900 GB/s for H100, 1800 GB/s for Blackwell).
Degraded: Links in "inactive" state, increasing replay count, bandwidth below spec = reseat GPU or replace cable/tray.
30
How do you validate InfiniBand fabric health?
▾
ibstat # local HCA status (port state, speed, GUID) ibstatus # quick link state check ibdiagnet # full fabric diagnostic (topology, errors, routing) ibping -G # ping remote HCA by GUID ib_write_bw # bandwidth test (RDMA write) ib_write_lat # latency test perfquery # per-port performance counters iblinkinfo # link width, speed, errors per port
Key checks: All ports LinkUp, speed matches (NDR=4x/HDR=4x), zero symbol errors, ibdiagnet clean report.
UFM provides GUI-based fabric health dashboard + automated alerts.
31
What GPU metrics indicate problems?
▾
Temperature: Throttle at ~83°C, shutdown at ~90°C. Sustained high = cooling issue.
ECC errors: Correctable (CE) increasing trend = failing memory. Uncorrectable (UCE) = immediate drain + RMA.
XID errors: GPU fault codes in dmesg/syslog. XID 79 = fallen off bus. XID 48 = double-bit ECC.
Clock throttling: SM clock below max = power/thermal throttle. Check nvidia-smi -q -d PERFORMANCE.
Retired pages: Pages retired due to ECC errors. High count = replace GPU.
PCIe errors: Replay count increasing = bad PCIe slot or riser.
32
ECC error handling: correctable vs uncorrectable?
▾
Correctable (CE / Single-bit): Hardware auto-corrects. Few per day = normal. Increasing trend = early warning of failing memory. Monitor via DCGM, set policy threshold.
Uncorrectable (UCE / Double-bit): Cannot be corrected. Causes job failure, data corruption risk. Action: drain node immediately, run DCGM r=4 diagnostics, RMA GPU.
Page retirement: NVIDIA driver auto-retires pages with repeated CE. Check with nvidia-smi -q -d RETIRED_PAGES. High retirement count = replace GPU.
Volatile vs Aggregate: Volatile = since last reset. Aggregate = lifetime total.
33
What dcgmi commands are essential for operations?
▾
dcgmi discovery # identify GPUs in system dcgmi group -l # list GPU groups dcgmi group -c "groupname" -a 0,1 # create group with GPU 0,1 dcgmi health -s mpi # set health watches (m=mem,p=pcie,i=inforom) dcgmi diag -r 3 # run level 3 diagnostics dcgmi dmon -e 203,204,1001,1002 # real-time monitoring (specific fields) dcgmi stats -e job123 # job-level statistics dcgmi topo -g 1 # topology for group 1 dcgmi config -g 1 --set # configure group settings dcgmi policy -g 1 --set 0,0 -e # set ECC error policy
Field IDs: 203=GPU temp, 204=power, 1001=GPU util, 1002=mem util, 1004=SM clock.
34
What is RDMA and why is it critical for AI clusters?
▾
Remote Direct Memory Access — read/write remote machine's memory directly, bypassing CPU and OS kernel.
Why critical: During distributed training, GPUs exchange gradients millions of times per step. RDMA does this at wire speed (~1μs latency on IB) vs ~10μs+ through kernel stack.
Types: InfiniBand native RDMA (best), RoCE (RDMA over Converged Ethernet — needs PFC/ECN), iWARP (TCP-based, worst performance).
GPU Direct RDMA: Network adapter reads/writes GPU memory directly — no CPU bounce buffer.
35
What is SHARP and how does it improve performance?
▾
Scalable Hierarchical Aggregation and Reduction Protocol — in-network computing. Reduction operations performed inside the switches, not on GPUs/CPUs.
How: Aggregation tree overlaid on fat-tree topology. Data enters at leaves, reduces at each aggregation node, result flows back.
Performance: 5x AllReduce improvement, up to 9x MPI Barrier. ~95% network bandwidth for AllReduce on HDR IB.
Quantum-X800: 14.4 TFLOPS in-network compute.
NCCL integration: NCCL auto-detects SHARP, offloads collectives. Reduces SM usage from 16+ to 6.
Operations: AllReduce, Reduce, Broadcast, AllGather, ReduceScatter (NCCL 2.27+).
36
What is adaptive routing in InfiniBand?
▾
Switch selects output port based on real-time port load (free adaptive routing).
Traditional IB: subnet manager computes static routes. Problem: hotspots when traffic patterns change.
Adaptive routing: Switch dynamically picks least-loaded path among valid routes. Spreads load across parallel spine links.
ARN (Adaptive Routing Notification): Switch reports congestion above threshold to upstream switches — cascading load awareness.
Combined with DCQCN congestion control for optimal fabric performance.
37
What is DCQCN congestion control?
▾
Data Center Quantized Congestion Notification. Combines ECN + PFC.
Flow: Switch detects congestion → marks CE bit (ECN) → Receiver sends CNP (Congestion Notification Packet) to sender → Sender reduces rate.
Sender behavior: Maintains "alpha" value per QP estimating congestion severity. Rate reduced on first CNP. Rate increases via timer and byte counter.
Key params: DcqcnRateToSetOnFirstCnp, DcqcnRateReduceMonitorPeriod, DcqcnRpgByteResetx64, DcqcnRpgTimeReset.
PFC (Priority Flow Control): Prevents packet drops by pausing sender. Last resort — sustained PFC = design problem.
38
GPUDirect RDMA vs GPUDirect Storage?
▾
GPUDirect RDMA: Network adapters access GPU memory directly (no CPU). Critical for training — gradient exchange at wire speed.
GPUDirect Storage (GDS): NVMe/storage transfers to GPU memory directly via DMA engine. Bypasses CPU bounce buffer.
GDS performance: Up to 3.5x higher bandwidth, 3.5x lower latency. DeepCAM: 6.6x speedup with GDS + DALI.
Both eliminate unnecessary data copies between CPU memory and GPU memory — the CPU is no longer in the data path.
39
What is the Magnum IO stack?
▾
Magnum IO = umbrella for all NVIDIA I/O acceleration technologies.
Components:
- GPUDirect Storage — NVMe ↔ GPU memory
- GPUDirect RDMA — Network ↔ GPU memory
- NCCL — GPU collective communications
- NVSHMEM — GPU-initiated one-sided communication (OpenSHMEM for GPUs)
- cuFile — API for GDS
Leverages InfiniBand + NVLink interconnects. Goal: remove CPU from every data path.
40
NCCL collective operations?
▾
AllReduce: Every GPU gets the reduced result (most important for training — gradient sync).
AllGather: Every GPU gets all data from all GPUs.
ReduceScatter: Reduce + scatter result chunks to different GPUs.
Broadcast: One GPU sends to all others.
Reduce: All data reduced to one GPU.
Send/Recv: Point-to-point communication.
NCCL is topology-aware — auto-detects NVLink, PCIe, IB, Ethernet and builds optimal communication tree. SHARP offloads AllReduce/Reduce to switches.
41
UFM tiers and capabilities?
▾
UFM Telemetry: Basic — bandwidth, latency, error rates. 120+ unique counters per port (BER, temp, histograms, retransmissions).
UFM Enterprise: Adds network validation, provisioning, congestion analysis, cable management, automated discovery, adaptive routing config, SHARP tree management.
UFM Cyber-AI: Adds AI/ML predictive analytics, anomaly detection, security threat detection, long-term trend analysis.
SuperPOD uses: UFM 3.5 Appliance, Enterprise Edition. Runs on dedicated node (one node per SU in H100 config).

▾

45
cmsh device states?
▾
[UP] — Monitored, operational. Normal healthy state.
[DOWN] — Monitored, unexpected failure. Alerts triggered.
[CLOSED](UP) — Intentionally up but mostly ignored by monitoring. Maintenance mode while still running.
[CLOSED](DOWN) — Intentionally down, ignored by monitoring. Planned shutdown/decommission.
Key: [UP]/[DOWN] = monitored. [CLOSED] = intentionally set by admin, not alerting.
46
How are nodes provisioned via BCM?
▾
PXE Boot: Node boots from network → DHCP from head node → TFTP loads kernel/initrd → Node pulls software image from head node → Installs to local disk or runs diskless.
Software images: Full Linux filesystem directories stored on head node. Clone and customize per node category.
Categories: Group nodes by function (gpu-compute, storage, management). Category defines: software image, roles (slurmclient, k8s-worker), network config.
Roles: Assign services to categories — assign slurmclient, assign kubernetes-worker.

49
Slurm configuration for GPU clusters?
▾
GRES (Generic Resources): Define GPUs in slurm.conf and gres.conf.
# gres.conf Name=gpu Type=h100 File=/dev/nvidia[0-7] # slurm.conf GresTypes=gpu NodeName=dgx[001-032] Gres=gpu:h100:8 # Job submission sbatch --gres=gpu:8 --nodes=4 train.sh srun --gres=gpu:4 --ntasks=4 nccl-tests

Key concepts: Partitions, QOS, fair-share scheduling, gang scheduling, preemption, --exclusive for full-node allocation.
50
What is Enroot + Pyxis?
▾
Enroot: NVIDIA's rootless container runtime. Runs containers without Docker daemon, no root required. Designed for HPC/Slurm environments.
Pyxis: Slurm SPANK plugin that integrates Enroot into Slurm. Allows --container-image flag on srun/sbatch.
Why not Docker for training? Docker requires root daemon, has overhead, doesn't integrate with Slurm scheduling natively. Enroot+Pyxis = lightweight, rootless, Slurm-native.
Docker is fine for Kubernetes/inference. Enroot+Pyxis is for Slurm-based training.

Domain 4: Troubleshoot & Optimize (12%)

Hardware faults, GPU replacement, performance tuning, storage optimization

54
Common XID errors and what they mean?
▾
XID = GPU fault code in kernel log (dmesg).
XID 13: Graphics Engine Exception — GPU compute error, usually software bug or unstable OC.
XID 31: GPU memory page fault — bad memory access in kernel.
XID 43: GPU stopped responding — hung GPU, timeout.
XID 45: Preemptive GPU cleanup — driver recovering from error.
XID 48: Double-bit ECC error — uncorrectable, drain + RMA.
XID 63: Row remapping failure — ECC bank exhausted.
XID 79: GPU has fallen off the bus — PCIe link failure, reseat GPU or replace riser.
XID 94: Contained ECC error (Hopper+) — error contained, job can continue.
Check: dmesg | grep -i xid or nvidia-smi -q -d ECC.
55
GPU replacement procedure?
▾
1. Drain node from scheduler: scontrol update nodename=dgx001 state=drain reason="GPU RMA"
2. Verify no running jobs on node
3. Set node to CLOSED(DOWN) in BCM
4. Power off via BMC or device power off
5. Physically replace GPU (ESD precautions, note serial number)
6. Power on, verify in BIOS
7. Boot OS, verify nvidia-smi shows new GPU
8. Run dcgmi diag -r 3 on node — validate replacement
9. Run single-node NCCL test — validate NVLink
10. Set node UP in BCM, undrain in Slurm
56
Performance tuning checklist?
▾
GPU: Persistence mode ON (nvidia-smi -pm 1). Clock speeds locked to max (nvidia-smi -lgc MAX_CLOCK). ECC enabled. Power limit at TDP.
CPU: Performance governor (cpupower frequency-set -g performance). NUMA-aware job placement. Disable C-states in BIOS.
Network: Adaptive routing enabled. SHARP enabled for collectives. PFC/ECN configured for RoCE if using Ethernet.
Storage: GPUDirect Storage enabled. IO scheduler = none for NVMe. Parallel FS tuned (stripe count, stripe size).
Software: Latest matched driver + CUDA + NCCL. NCCL env vars: NCCL_IB_HCA, NCCL_SOCKET_IFNAME, NCCL_ALGO.

Storage performance troubleshooting?
▾
Symptoms: GPU utilization drops during data loading. Training throughput below expected.
Check:
1. nvidia-smi dmon — if GPU util spiky/low, data pipeline is the bottleneck
2. iostat / sar — disk/NFS throughput
3. Storage fabric bandwidth — IB counters via UFM or perfquery
4. GPUDirect Storage enabled? gdscheck -p
5. Lustre/GPFS stripe settings — match stripe count to OST count
6. DataLoader workers — increase num_workers in PyTorch
7. Cache: use local NVMe as staging/cache tier
Target: 40+ GBps per node sustained throughput.
58
InfiniBand link troubleshooting?
▾
Link down: ibstat shows port "Down" → check cable seating, try different port, replace cable/transceiver.
Link width reduced: Should be 4x (HDR) or 4x (NDR). If 1x → bad cable or connector damage.
High symbol error rate: perfquery shows increasing errors → replace cable (most common fix).
BER (Bit Error Rate): UFM telemetry. Pre-FEC BER increasing = cable degrading.
Congestion: UFM congestion map → check for hotspot → add adaptive routing or rebalance topology.
Speed negotiation: Port shows lower speed than expected → firmware mismatch or incompatible transceiver.

Docker GPU troubleshooting?

→ nvidia-container-toolkit not installed or runtime not configured.

"Failed to initialize NVML: Unknown Error":

BlueField three modes of operation?
▾
1. NIC Mode: Arm cores inactive. Functions as standard ConnectX adapter. Full host access.
mlxconfig -d /dev/mst/<dev> s INTERNAL_CPU_OFFLOAD_ENGINE=1
2. DPU Mode (ECPF — Default): Arm cores active, control NIC resources. Embedded switch managed by Arm. Host has restricted access. OpenSM runs from Arm side.
mlxconfig -d /dev/mst/<dev> s INTERNAL_CPU_OFFLOAD_ENGINE=0
3. Zero Trust / Restricted Mode: Host fully isolated. Port ownership, counters, RShim, firmware flashing all DISABLED from host.
mlxprivhost -d /dev/mst/<dev> r --disable_rshim --disable_tracer --disable_counter_rd --disable_port_owner
Note: Separated mode is OBSOLETE for BlueField-3.

MIG configuration: single vs mixed strategy?
▾
MIG = Multi-Instance GPU. Hardware partitioning into up to 7 isolated instances. A100/H100 only.

Single Strategy:
- All GPUs on node have identical MIG geometry
- Exposed as standard nvidia.com/gpu resources
- Label: nvidia.com/gpu.product=NVIDIA-H100-80GB-HBM3-MIG-1g.10gb
- Simpler scheduling

Mixed Strategy:
- Each GPU can have different MIG geometry
- Resources per-profile: nvidia.com/mig-1g.5gb, nvidia.com/mig-3g.20gb
- Configured via default-mig-parted-config ConfigMap
- More flexible but complex scheduling

nvidia-smi mig -cgi 9,9,9,9,9,9,9 -C # create 7x 1g.10gb nvidia-smi mig -lgi # list GPU instances nvidia-smi mig -dci && nvidia-smi mig -dgi # destroy all
65
Data parallelism vs model parallelism vs pipeline parallelism?
▾
Data Parallelism: Same model on every GPU, different data batches. Sync gradients via AllReduce (NCCL). Works when model fits one GPU.
Model (Tensor) Parallelism: Model split across GPUs — different parts of same layer on different GPUs. Needs NVLink bandwidth (intra-node). For very wide layers.
Pipeline Parallelism: Different layers on different GPUs. Forward pass flows through pipeline stages. Micro-batching to fill pipeline. Inter-node friendly (less bandwidth needed).
In practice: Large models use all three (3D parallelism). Data parallel across nodes, tensor parallel within node (NVLink), pipeline parallel across node groups.

What is persistence mode and why enable it?
▾
Persistence mode keeps the NVIDIA driver loaded even when no GPU application is running.
Without it: Driver unloads between jobs → next job has 1-3 second cold-start delay while driver reinitializes.
With it: Driver stays resident → instant job start, consistent monitoring (DCGM needs it), stable clock settings.
nvidia-smi -pm 1 # enable nvidia-smi -pm 0 # disable # OR use nvidia-persistenced daemon
Always enable in production. DCGM diagnostics check for this (Deployment plugin).
67
What is page retirement?
▾
GPU driver auto-retires memory pages that have repeated ECC errors.
Single-bit retirement: Page had multiple correctable errors. Retired at next GPU reset. Memory still usable minus that page.
Double-bit retirement: Page had uncorrectable error. Retired immediately.
Check: nvidia-smi -q -d RETIRED_PAGES
Thresholds: If retired page count gets high (varies by GPU, typically >60), GPU is failing → RMA.
Pending retirements: Some retirements require GPU reset to take effect. nvidia-smi -q -d RETIRED_PAGES shows "pending" status.

What is NVSHMEM?
▾

Difference from NCCL: NCCL = collective operations (AllReduce, AllGather). NVSHMEM = point-to-point put/get operations initiated directly from GPU kernels.
Use case: Fine-grained communication patterns where GPUs need to read/write specific remote memory locations without synchronizing all GPUs. Useful for sparse communication, graph algorithms, irregular access patterns.
Part of Magnum IO stack.
