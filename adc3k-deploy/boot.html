AI FACTORY BOOT SEQUENCE

How You Turn On an NVIDIA AI Factory — Step by Step
Based on NVIDIA NCP-AIO Training Outline + AI Factory Design Guide

The Big Picture

An AI factory isn't one thing you power on. It's 13 layers you bring up in order — site prep and credentials first, hardware second, networking third, management and workloads last.

Skip a step or do them out of order and the whole thing breaks. This is the sequence.

00a
Site Survey & ReadinessBefore you buy a single GPU
  • Power capacity assessment — how many megawatts available? What's the utility interconnect agreement? Transformer capacity? A single SuperPOD can pull 3-5 MW. You need commitments from the utility before signing hardware POs.
  • Cooling capacity — water supply (volume, temperature, quality) for cooling towers or dry coolers. Immersion tanks need fluid procurement (EC-110, 3M Novec, etc). CDU placement planning. Heat rejection strategy (where does the waste heat go?).
  • Network connectivity — fiber to the building. Dark fiber or lit service? Latency to nearest peering point. Redundant paths from different carriers. This determines tenant experience.
  • Physical space — floor load capacity (NVL72 racks are heavy), ceiling height (for overhead cabling/cooling), raised floor vs slab. Clear path for forklift delivery of racks.
  • Environmental — permits, zoning (industrial use), noise ordinances (cooling equipment), environmental impact (water discharge). Louisiana: check DEQ, Army Corps if near waterways.
  • Redundancy design — N+1 or 2N power? Generator fuel storage and runtime calculations. UPS sizing. Fire suppression (clean agent, not water, around GPU racks).
  • Network architecture — design the InfiniBand fabric topology (fat-tree, rail-optimized). How many spine/leaf switches? Port count math determines rack count limits.
  • This is the make-or-break step. A bad site kills the project. Willow Glen scores 94 because power, water, fiber, and space are all pre-existing at the former Entergy station.

00b
Credentials, Licenses & KeysYou can't install anything without these
  • NGC API Key — required to pull containers, models, and Helm charts from NVIDIA's NGC catalog. Generated at ngc.nvidia.com. Without this, you can't deploy any NVIDIA AI software. Every node that pulls containers needs it.
  • NVIDIA AI Enterprise (NVAIE) License — the enterprise software subscription. Unlocks support, security patches, certified containers, Run:ai, and production-grade tools. Per-GPU annual license. This is NVIDIA's recurring revenue model — hardware is one-time, software is annual.
  • Base Command Manager (BCM) License — included with DGX systems or purchased separately for third-party servers (Supermicro, Dell, etc). Required to manage the cluster. No BCM = no centralized management.
  • Run:ai License — comes with NVAIE or purchased standalone. Required for GPU orchestration and multi-tenancy. The control plane needs license activation before it schedules anything.
  • InfiniBand UFM License — Unified Fabric Manager. Base version may be included with switches; advanced features (telemetry, SHARP management, cyber security) require UFM Enterprise license.
  • NVIDIA Inception membership — free program for startups. Gets you cloud credits, technical support, co-marketing, investor connects, and discounts on DGX. Get ADC registered at GTC — Inception desk on the expo floor.
  • NVentures relationship — NVIDIA's VC arm. Not a license, but a strategic relationship. They invest in companies building on the NVIDIA stack. Wednesday 2 PM at GTC.
  • SSH keys & certificates — generate keypairs for all admin accounts before cluster bring-up. Root access to head node, BMC/IPMI credentials for every server, switch admin passwords. Store in a vault (HashiCorp Vault, not a spreadsheet).
  • DNS & TLS certificates — internal DNS for node naming (gpu-node-001.wg.adc3k.local). TLS certs for dashboards, APIs, and management interfaces. Wildcard cert or per-service from internal CA.
  • Monitoring credentials — Prometheus, Grafana, Alertmanager. Service accounts for DCGM exporters, UFM telemetry, BCM health checks. These feed your NOC dashboards.
  • Missing one key or expired license can halt an entire deployment. Get all of these sorted BEFORE hardware arrives. Put them in a secure vault with access controls.

01
Physical InfrastructurePower + Cooling + Rack & Stack
  • Power delivery — redundant feeds, PDUs, UPS. An NVL72 rack pulls 120+ kW. A SuperPOD needs megawatts.
  • Cooling loops online — CDUs pumping, immersion tanks filled (EC-110), cold plates seated. Nothing powers on until cooling is verified.
  • Racks mounted — servers, switches, cabling. InfiniBand cables connected (passive copper for short runs, active optical for longer).
  • ADC 3-layer power stack — nat gas + solar (primary), grid (backup), diesel gensets (emergency). All three verified before compute powers on.
  • Out-of-band management — BMC/IPMI network up so you can reach every node remotely even if the main network is down.
|
02
GPU & Compute HardwareDrivers, NVSM, DCGM
  • Install GPU drivers — NVIDIA driver package on every compute node. Must match your GPU generation (Blackwell, Hopper, etc).
  • NVSM (NVIDIA System Management) — validates server hardware health: GPUs, NVLink, PCIe, memory, fans, thermals. First sanity check.
  • DCGM (Data Center GPU Manager) — real-time telemetry on every GPU. Temperature, power draw, ECC errors, NVLink status. Feeds into Prometheus/Grafana.
  • dcgmi diag -r 3 — run Level 3 diagnostics. Tests GPU memory, compute, NVLink bandwidth. Catches bad GPUs before workloads hit them.
  • nvidia-smi — quick CLI check. Shows GPU model, driver version, memory usage, temperature, power draw, running processes.
  • Every GPU must pass diagnostics before moving to Step 3. One bad GPU in a training run can corrupt the entire job.
|
03
Networking — InfiniBand FabricQuantum Switches + UFM
  • InfiniBand spine — Quantum-2 switches (400 Gb/s per port). This is the backbone connecting every rack in the building.
  • UFM (Unified Fabric Manager) — discovers the fabric topology, assigns LIDs (local IDs), manages routing, monitors health. The "brain" of the network.
  • SHARP (Scalable Hierarchical Aggregation and Reduction Protocol) — enables in-network compute. Math operations happen on the switch before data reaches the GPU. Reduces traffic by up to 2x.
  • Subnet Manager — runs on UFM. Manages all InfiniBand addressing and routing. One primary, one standby for failover.
  • ibstat / ibdiagnet — verify link status, check for errors, validate fabric health.
  • NVLink mesh — GPU-to-GPU within a node (and across nodes in NVL72). 900 GB/s per GPU. Verified separately from InfiniBand.
  • The network is the computer. If InfiniBand has errors, training jobs will hang or produce garbage results.
|
04
BlueField DPUsInfrastructure on a Chip
  • BlueField bring-up — each DPU runs its own ARM-based OS (DOCA). Boot it, configure networking, verify connectivity.
  • What it offloads — networking, encryption, firewall, storage I/O, RDMA, microsegmentation. All handled on the DPU so the GPU never touches it.
  • BlueMan service — management daemon running on the DPU. Handles provisioning, monitoring, firmware updates.
  • Zero-trust security — DPU enforces network policies at the hardware level. Even if a VM or container is compromised, the DPU controls what traffic gets in/out.
  • GPUDirect RDMA — DPU enables network data to flow directly to GPU memory, bypassing CPU. Critical for distributed training performance.
  • DPUs are what separate an AI factory from a regular server farm. They turn "dumb NICs" into programmable infrastructure.
|
05
StorageParallel File Systems + GPUDirect
  • Parallel file system — Lustre, GPFS (Spectrum Scale), or WekaFS. Data striped across hundreds of NVMe drives for maximum throughput.
  • GPUDirect Storage — data flows straight from NVMe to GPU memory, bypassing CPU. Eliminates the biggest bottleneck in the data pipeline.
  • Performance testing — benchmark read/write speeds before running workloads. If storage can't feed GPUs fast enough, expensive GPUs sit idle waiting for data.
  • Tiered storage — hot tier (NVMe, fast), warm tier (SSD), cold tier (HDD/object storage). Training data on hot, checkpoints on warm, archives on cold.
  • Capacity planning — a single LLM training run can generate terabytes of checkpoints. Plan for it or your storage fills up mid-training.
  • The #1 performance killer in AI training isn't slow GPUs — it's slow storage. GPUs can consume data faster than most storage systems can deliver it.
|
06
Base Command Manager (BCM)The Foreman
  • Head node install — BCM lives on the head node. This is the management server that controls everything else.
  • Cluster bring-up — PXE boot compute nodes from head node. BCM pushes OS images, drivers, and configuration automatically.
  • cmsh (Cluster Management Shell) — CLI for everything. Add nodes, assign roles, manage images, check health. device list, device status.
  • Node categories — group nodes by role (compute, storage, login, management). Each category gets its own software image and config.
  • Software images — BCM manages golden images for each node type. Update the image, push to all nodes in that category at once.
  • User management — LDAP integration, SSH keys, role-based access. Control who can submit jobs and to which resources.
  • Monitoring — BCM collects health from every node, switch, and GPU. Alerts on failures, thermals, hardware errors.
  • BCM is the foundation. Everything above it — Slurm, Kubernetes, Run:ai — depends on BCM having the cluster healthy and provisioned.
|
07
SlurmHPC Job Scheduler
  • Install Slurm — BCM assigns the slurmclient role to compute nodes. Head node runs the Slurm controller.
  • Partitions — divide the cluster into queues (partitions). E.g., "training" (big GPU jobs), "inference" (small fast jobs), "debug" (testing).
  • Job submission — sbatch for batch jobs, srun for interactive, salloc for resource allocation. Users submit, Slurm schedules.
  • GRES (Generic Resources) — how Slurm tracks GPUs. --gres=gpu:4 requests 4 GPUs for a job.
  • Fair-share scheduling — prevents one user from hogging all GPUs. Balances resources across teams/tenants.
  • Slurm is the traditional way. Still used for tightly coupled HPC workloads (CFD, molecular dynamics, seismic). NVIDIA now owns SchedMD (Slurm developer).
|
08
KubernetesContainer Orchestration
  • K8s cluster — control plane on management nodes, worker nodes on compute. BCM handles the underlying OS, K8s handles the containers.
  • GPU Operator — auto-installs NVIDIA drivers, container toolkit, device plugins, and DCGM inside K8s. Drop a node in, it becomes GPU-ready automatically.
  • Device plugin — exposes GPUs to Kubernetes. Pods request GPUs like any other resource: nvidia.com/gpu: 2.
  • Container toolkit — lets containers access the GPU driver. The bridge between Docker/containerd and NVIDIA hardware.
  • Namespaces — isolate tenants. Team A's workloads can't see or affect Team B's. Each namespace gets resource quotas.
  • Kubernetes is the foundation for Run:ai and modern AI workloads. Most AI frameworks ship as containers — K8s is how you run them at scale.
|
09
Run:aiGPU Orchestration & Multi-Tenancy
  • Sits on top of Kubernetes — Run:ai adds GPU-aware scheduling that K8s doesn't have natively.
  • Dynamic GPU Fractions — split one GPU across multiple workloads, or gang multiple GPUs for one job. No GPU sits idle.
  • Projects & Departments — organize tenants. Each gets a GPU quota with guaranteed minimums and borrowable maximums.
  • Fairness policies — if Team A isn't using their GPUs, Team B can borrow them. When Team A needs them back, Run:ai preempts gracefully.
  • Workload types — Training (long, multi-GPU), Interactive (Jupyter notebooks), Inference (serving endpoints). Each scheduled differently.
  • Dashboard — visual GPU utilization across the entire cluster. See who's using what, what's idle, what's queued.
  • This is where the money is. Run:ai can push GPU utilization from 30-40% (typical) to 80%+. That's 2x the revenue from the same hardware.
|
10
GPU Partitioning (MIG)Slice GPUs for Shared Use
  • MIG (Multi-Instance GPU) — hardware-level GPU partitioning. One A100/H100 becomes up to 7 independent GPU instances.
  • Each instance gets its own compute, memory, and cache. Fully isolated — one tenant's workload can't see or affect another's.
  • Use case — inference workloads that don't need a full GPU. Serve 7 different models on one physical GPU.
  • nvidia-smi mig — create, list, and destroy MIG instances. Configure profiles (1g.5gb, 2g.10gb, 3g.20gb, etc).
  • Run:ai + MIG — Run:ai can schedule workloads to MIG instances automatically. Combines software scheduling with hardware isolation.
  • MIG + Run:ai fractions = maximum GPU utilization. Hardware isolation where you need security, software fractions where you need flexibility.
|
11
AI Software & NGCPull Containers, Run Workloads
  • NGC Catalog — NVIDIA's container registry. Pre-optimized containers for PyTorch, TensorFlow, NeMo, Triton, RAPIDS, and hundreds more.
  • Triton Inference Server — serves trained models in production. Handles batching, model versioning, multi-framework support. This is your inference engine.
  • NeMo — framework for building and fine-tuning LLMs. Training, RLHF, guardrails, deployment — end to end.
  • RAPIDS — GPU-accelerated data science. Pandas, scikit-learn, graph analytics — all on GPU instead of CPU. 10-100x faster.
  • ngc registry image list — browse available containers. docker pull nvcr.io/nvidia/pytorch:xx.xx-py3 to grab one.
  • NVIDIA AI Enterprise — the enterprise software suite. Includes all the above plus support, security patches, and certified containers.
  • This is where the factory starts producing. Everything before this was building the factory. Now it's running.

The One-Liner Version

Power & cool it → verify the GPUs → wire the fabric → bring up DPUs → mount storage → install BCM → add Slurm → layer Kubernetes → deploy Run:ai → partition GPUs → pull containers and run

Bottom up: hardware, then networking, then management, then scheduling, then workloads. Each layer depends on every layer below it.

GlossaryQuick-reference — tap a term, get the answer

SHARP
Scalable Hierarchical Aggregation and Reduction Protocol. InfiniBand switches do math operations (all-reduce, broadcast) right on the switch silicon instead of making GPUs do it. Cuts network traffic up to 2x. Built into Quantum-2 switches — not software you install.
MIG
Multi-Instance GPU. Hardware-level GPU partitioning. Splits one physical GPU into up to 7 independent instances, each with its own compute, memory, and cache. Fully isolated — one tenant can't see another. Used for inference workloads that don't need a full GPU.
BCM
Base Command Manager. NVIDIA's cluster management platform. Provisions bare metal, deploys OS images, manages firmware, assigns node roles. CLI is cmsh (cluster management shell). The foundation — everything else sits on top of it.
DCGM
Data Center GPU Manager. Real-time monitoring for every GPU — temperature, power draw, ECC errors, NVLink status, utilization. Feeds into Prometheus/Grafana. dcgmi diag runs health checks. If a GPU is going bad, DCGM catches it first.
DPU (BlueField)
Data Processing Unit. A full ARM-based computer on a network card. Offloads networking, security, encryption, firewall, storage I/O from the CPU/GPU. Runs its own OS (DOCA). Enables zero-trust security at hardware level. "Infrastructure on a chip."
DOCA
Data Center Infrastructure on a Chip Architecture. The SDK and runtime that runs on BlueField DPUs. Think of it as the operating system for the DPU — lets you program what the DPU offloads (firewall rules, encryption, storage, network policies).
UFM
Unified Fabric Manager. The brain of the InfiniBand network. Discovers topology, assigns addresses (LIDs), manages routing, monitors health, enables SHARP. Runs the Subnet Manager. One primary + one standby for failover.
NVLink
GPU-to-GPU interconnect. 900 GB/s per GPU (Blackwell). Connects GPUs within a node and across nodes in NVL72 racks. NVLink Switch manages the topology. This is how 72 GPUs act like one giant GPU. Way faster than PCIe.
NVL72
72-GPU rack. A single rack with 72 Blackwell GPUs connected via NVLink. Pulls 120+ kW. Requires liquid cooling. The building block of a SuperPOD. All 72 GPUs share memory and compute as one unified system.
SuperPOD
NVIDIA's reference AI factory architecture. Multiple NVL72 racks connected by InfiniBand spine. Managed by BCM. The full-scale deployment — what Willow Glen is being designed around. Scales from hundreds to tens of thousands of GPUs.
InfiniBand
High-speed interconnect fabric. 400 Gb/s per port (Quantum-2). Ultra-low latency, RDMA-capable. Not just a network — supports in-network compute (SHARP). The backbone connecting every rack in an AI factory. Managed by UFM.
RDMA
Remote Direct Memory Access. Lets one machine read/write another machine's memory directly, bypassing the CPU and OS kernel on both sides. Zero-copy, ultra-low latency. This is why InfiniBand is faster than Ethernet for AI training.
GPUDirect Storage
Data path from NVMe drives straight to GPU memory. Bypasses CPU entirely. The storage equivalent of RDMA. Eliminates the CPU bottleneck when loading training data. Critical for keeping GPUs fed.
GPUDirect RDMA
Network data flows directly to GPU memory. Bypasses CPU. Enabled by BlueField DPU. Essential for distributed training where GPUs on different nodes exchange gradients at high speed.
Run:ai
GPU orchestration platform. Kubernetes-native. Dynamic GPU Fractions — splits or gangs GPUs per workload. Multi-tenant scheduling, fair-share policies, priority queues. Pushes GPU utilization from ~30% to 80%+. NVIDIA acquired Run:ai in 2024.
Slurm
HPC job scheduler. Traditional batch queue system. Submit jobs, Slurm allocates nodes and GPUs. Partitions, fair-share, GRES (GPU resources). NVIDIA acquired SchedMD (Slurm developer) Dec 2025. Still used for tightly coupled HPC workloads.
GPU Operator
Kubernetes operator for NVIDIA GPUs. Auto-installs drivers, container toolkit, device plugins, DCGM inside K8s clusters. Drop a new node in, GPU Operator makes it GPU-ready automatically. No manual driver installs.
NGC
NVIDIA GPU Cloud Catalog. Container registry with pre-optimized AI containers — PyTorch, TensorFlow, NeMo, Triton, RAPIDS. Pull and run. Already tuned for NVIDIA hardware. Also hosts pre-trained models and Helm charts.
Triton
NVIDIA Triton Inference Server. Serves trained models in production. Handles dynamic batching, model versioning, multi-framework (PyTorch, TensorFlow, ONNX). Your inference engine — this is how tenants deploy models as API endpoints.
NeMo
NVIDIA's LLM framework. End-to-end: data prep, training, fine-tuning, RLHF, guardrails (NeMo Guardrails), deployment. Built for enterprise-scale LLM development on NVIDIA hardware.
NVSM
NVIDIA System Management. Validates server hardware health — GPUs, NVLink, PCIe, memory, fans, thermals. First sanity check on new hardware before anything else runs. Different from DCGM (NVSM = hardware validation, DCGM = ongoing monitoring).
NCCL
NVIDIA Collective Communications Library. Handles multi-GPU and multi-node communication for distributed training — all-reduce, all-gather, broadcast. Works over NVLink (within node) and InfiniBand (across nodes). The glue for distributed training.
ECC
Error Correcting Code. Memory error detection and correction built into GPU HBM. Single-bit errors get corrected automatically. Multi-bit errors get flagged. DCGM monitors ECC counts — rising errors mean a GPU is degrading and needs replacement.
HBM
High Bandwidth Memory. Memory stacked directly on the GPU package. HBM3e: up to 288 GB at 8 TB/s bandwidth per GPU. HBM4 coming (Samsung, Booth #1207 at GTC). This is where model weights live — the GPU's fuel tank.
CDU
Coolant Distribution Unit. Pumps liquid coolant to server cold plates or immersion tanks, removes heat via heat exchangers. The heart of any liquid cooling system. Moves heat from the chips to where it can be rejected (or reused).
PFLOPS
PetaFLOPS. One quadrillion (10^15) floating-point operations per second. A single Blackwell GPU delivers ~20 PFLOPS of AI compute (FP4/FP8). A SuperPOD delivers exaflops. This is how you measure AI factory horsepower.
Inference vs Training
Training = teaching the model (expensive, long, lots of GPUs). Inference = using the trained model to get answers (cheaper, fast, revenue-generating). Training is a cost. Inference is a product. AI factories do both.
Accelerated Computing
The new term that replaces "HPC." GPU-accelerated everything — AI, simulation, rendering, analytics, scientific computing. The entire infrastructure is accelerated, not just the GPUs. Every chip (DPU, switch, NVLink) removes overhead so GPUs do nothing but compute.
NGC API Key
Authentication token for NVIDIA's container registry. Generated at ngc.nvidia.com. Required to pull any container, model, or Helm chart. Without it, you can't deploy NVIDIA AI software. Treat it like a password — store in a vault, rotate regularly.
NVAIE
NVIDIA AI Enterprise. The enterprise software subscription. Per-GPU annual license. Unlocks Run:ai, certified NGC containers, security patches, enterprise support, and production-grade tools. This is NVIDIA's recurring revenue model — hardware once, software forever.
NVIDIA Inception
Free startup program. Cloud credits, technical support, co-marketing, investor introductions, hardware discounts. No cost to join. Apply online or register at the Inception desk on the GTC expo floor. ADC qualifies — building on full NVIDIA stack.
NVentures
NVIDIA's corporate venture capital arm. Invests in companies building on the NVIDIA platform. Strategic money — comes with NVIDIA engineering support and market access, not just capital. Target: Wed Mar 18, 2 PM at GTC.
BMC / IPMI
Baseboard Management Controller / Intelligent Platform Management Interface. Out-of-band management built into every server. Lets you power on/off, access console, check hardware health remotely — even when the OS is down. Your lifeline when something breaks at 3 AM.
PXE Boot
Preboot Execution Environment. Boots servers over the network from the BCM head node. No USB drives, no local install media. BCM pushes the OS image to bare metal nodes automatically. This is how you provision 100+ nodes without touching each one.
PDU
Power Distribution Unit. Distributes power to racks. Smart PDUs monitor per-outlet power draw, enable remote power cycling, and feed data to monitoring dashboards. You need to know exactly how much power each rack pulls.
Fat-Tree Topology
Standard InfiniBand network design. Leaf switches connect to servers, spine switches connect leaf switches. Every node can talk to every other node at full bandwidth. Non-blocking — no bottlenecks. This is the SuperPOD reference architecture.
LID
Local Identifier. The InfiniBand equivalent of an IP address. Assigned by the Subnet Manager (UFM). Every port on every switch and HCA gets a unique LID. The fabric uses LIDs for routing. If LIDs are wrong, nothing talks.
GRES
Generic Resources (Slurm). How Slurm tracks GPUs. Jobs request GPUs with --gres=gpu:4. Slurm matches the request to available nodes. Without GRES configured, Slurm doesn't know GPUs exist.
K8s / Kubernetes
Container orchestration platform. Manages containerized workloads at scale. Control plane schedules pods to worker nodes. NVIDIA GPU Operator makes K8s GPU-aware. Run:ai sits on top of K8s for GPU-specific scheduling. The industry standard for deploying AI at scale.
RLHF
Reinforcement Learning from Human Feedback. How LLMs like Claude and ChatGPT get fine-tuned after initial training. Humans rate model outputs, model learns from the ratings. NeMo framework handles this on NVIDIA hardware.
